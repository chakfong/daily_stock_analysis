## 上下文

当前持仓交易录入的完整调用链分析（详见探索阶段分析）：

```
用户点击"提交交易"
  ├── POST /api/v1/portfolio/trades    ← 写入交易
  │     ├── 验证去重 (DB查询)
  │     ├── 卖方向: 加载全部交易+公司行为 (全量扫描)
  │     ├── INSERT 交易
  │     └── DELETE 全部缓存 (Position/Lot/Snapshot)
  │
  ├── GET /api/v1/portfolio/snapshot   ← 刷新快照 (全量回放#1)
  │     ├── SELECT 全部交易 WHERE date <= as_of
  │     ├── SELECT 全部资金流水
  │     ├── SELECT 全部公司行为
  │     ├── 合并排序 → 从最早事件逐笔回放
  │     ├── 每个持仓: 外部API取实时价 (efinance→akshare×3 fallback)
  │     └── 每个汇率转换: DB查FX rate
  │
  ├── GET /api/v1/portfolio/risk       ← 刷新风险 (内部又做全量回放#2)
  │     └── get_portfolio_snapshot() → 重复上述全部
  │
  └── GET /api/v1/portfolio/trades     ← 刷新流水列表
```

**核心问题**: 每次录入一笔交易都做 O(n) 的全量历史回放（n=历史事件总数），且执行两次。同时串行调用外部行情API。

**关键约束**:
- SQLite 数据库，写入串行化
- 现有 `PortfolioPosition`/`PortfolioPositionLot`/`PortfolioDailySnapshot` 缓存表已存在
- Python 3.14 + FastAPI + React/TypeScript 前端
- 成本计算方法: FIFO 和 AVG（加权平均）

## 目标 / 非目标

**目标：**
1. 将单次交易录入的 HTTP 请求从 4 个减少到 1 个
2. 消除全量历史回放，改为增量计算（O(1) 而非 O(n)）
3. 消除交易录入路径上的外部 API 调用（实时行情）
4. 消除重复的 Snapshot 计算
5. 确保 FIFO 和 AVG 两种成本方法的行为一致性

**非目标：**
- 不改变数据库 schema
- 不改变前端 UI 布局
- 不修改 CSV 导入流程
- 不修改资金流水和公司行为的录入流程（可后续扩展）
- 不引入 Redis/消息队列等新基础设施

## 决策

### 决策 1: 增量模式 vs 部分回放

**选择**: 直接增量应用（基于缓存表）

**替代方案**:
- A) 部分回放：记录上次缓存日期，只回放从该日期起的事件。优点是逻辑改动小，缺点是需要维护额外的"缓存截止日期"元数据，且回放仍需跨多个表。
- B) 直接增量应用：读取缓存持仓 → 应用单笔交易变更 → 写回缓存。优点是 O(1) 复杂度，不需要元数据；缺点是需要确保增量逻辑与全量回放逻辑等价。

选择 B 的原因：缓存表已经包含完整的回放结果状态，增量应用买入/卖出只需要操作对应的 symbol 的 lot 和 position，逻辑清晰可验证。且可以与现有的 `_consume_fifo_lots` 和 AVG 计算逻辑复用。

### 决策 2: 合并接口 vs 后端事件驱动

**选择**: 新增合并接口 `POST /trades?return_snapshot=true`

**替代方案**:
- A) WebSocket 推送：后端写入后推送更新。复杂度高，前端需要管理 WS 连接状态。
- B) 合并接口：trade POST 响应中直接携带更新后的 snapshot。简单可靠，一次请求完成。

选择 B 的原因：SQLite 单写入者模型下，在同一个写事务中完成 "写入+更新缓存+返回结果" 是最高效的方式。不引入额外的通信复杂度。

### 决策 3: Fast 模式作为参数 vs 独立接口

**选择**: 在 snapshot 查询中增加 `fast` 参数

**替代方案**:
- A) 独立 `/snapshot-fast` 接口。增加路由数量，维护成本高。
- B) `fast` 查询参数。前端可以按场景控制（录入后 fast=true，手动刷新 fast=false）。

选择 B 的原因：语义清晰，不影响现有调用方，便于渐进式采用。

### 决策 4: Risk 复用 Snapshot

**选择**: `PortfolioRiskService` 新增 `get_risk_report_from_snapshot()` 方法

前端在 `include_risk=true` 时，服务端先计算 snapshot，再将结果传入 risk service，避免 risk 内部重新计算 snapshot。

### 决策 5: 前端请求策略

**选择**: 录入后不发 risk、不发 trades list

- Risk 数据（集中度/回撤/止损）不会因为单笔交易而发生剧烈变化，无需立即刷新
- Trades 列表采用前端乐观插入：将新交易直接 prepend 到列表头部

## 风险 / 权衡

- **[同一天多事件排序]**: 如果用户在同一个交易日先录入了一笔买入、再录入一笔卖出，增量应用必须正确处理顺序（现金→公司行为→交易）。当前设计假设同一天的增量操作基于缓存状态，不再重新排序当天所有事件 → **缓解**: 如果缓存状态正确反映了已录入的事件（按时序），后续增量应用也是正确的。对于同一天先买后卖的场景：第一笔买入增量更新了缓存，第二笔卖出基于更新后的缓存验证可用数量，结果是正确的。

- **[FIFO Lot 精确性]**: 增量模式下，卖出时消费 FIFO lot 的行为必须与全量回放中 `_consume_fifo_lots` 的行为完全一致 → **缓解**: 复用现有的 `_consume_fifo_lots` 方法，从缓存中加载 lot 列表，执行消费后写回。

- **[向前兼容]**: 如果缓存在某次全量回放时被重建（例如成本方法切换），增量写入需要能正确处理"空缓存"的情况 → **缓解**: 如果缓存为空（首次使用或缓存被清理），回退到全量回放模式。

- **[事务边界]**: 增量模式的 "读缓存→修改→写回" 必须在同一个 `portfolio_write_session` 事务中完成 → **缓解**: 利用现有的 `BEGIN IMMEDIATE` 事务机制，整个流程在一个 session 内完成。

## 迁移计划

1. **向后兼容**: 新接口与旧接口共存。前端可以逐步迁移，旧行为不受影响。
2. **部署步骤**:
   - 阶段1: 部署后端新接口（`POST /trades?return_snapshot=true`, `GET /snapshot?fast=true`, `GET /snapshot?include_risk=true`）
   - 阶段2: 前端切换到新接口
   - 阶段3: 观察无问题后，标记旧接口为 deprecated（可选）
3. **回滚**: 前端切换回旧的 4 请求模式即可，后端无需回滚（新接口不影响旧接口）
4. **测试策略**:
   - 单元测试: 增量引擎 vs 全量回放的一致性验证（相同输入→相同输出）
   - 集成测试: API 端到端测试
   - 手动测试: 买入、卖出、超卖验证、FIFO/AVG 双模式
